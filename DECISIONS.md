## 2026-03-26: Gerenciador de pacotes

**Decisão**: `pnpm >= 10.x`
**Motivo**: Armazenamento eficiente via hard links (único depósito central), isolamento correto de dependências, lockfile estável entre máquinas e suporte nativo na Vercel.
**Alternativas descartadas**:
- `npm` — lento, ocupa mais disco, instala em cada projeto separadamente
- `yarn` — perdeu diferencial após pnpm amadurecer
- `bun` — runtime ainda imaturo para SvelteKit/Drizzle em produção
**Trade-off aceito**: Colegas precisam ter pnpm instalado para contribuir.
**Revisável quando**: Bun estabilizar suporte completo ao ecossistema SvelteKit + Drizzle.

## 2026-03-26: Versão Node.js

**Decisão**: `Node.js >= 24.x` (Active LTS)
**Motivo**: Versão com suporte ativo mais longo (EOL 2028), padrão atual da Vercel, compatível com todo o stack.
**Alternativas descartadas**:
- Node 20 — EOL em 30/04/2026, sem sentido em projetos novos
- Node 22 — Maintenance LTS, EOL em 2027
- Node 25 — ímpar, não vira LTS
**Trade-off aceito**: Nenhum relevante para o stack atual.
**Revisável quando**: Node 26 entrar em Active LTS (previsão: outubro/2026).

## 2026-03-26: Framework frontend

**Decisão**: `SvelteKit >= 2.5` com `Svelte 5 >= 5.5` (runes)
**Motivo**: SSR nativo, roteamento por arquivos, form actions com progressive enhancement, runes tornam o estado mais previsível que stores.
**Alternativas descartadas**:
- Next.js — React tem overhead maior, ecossistema mais complexo para o perfil do projeto
- Astro — excelente para conteúdo estático, mas sistemas CRUD precisam de reatividade plena
**Trade-off aceito**: Ecossistema menor que React; alguns componentes precisam ser construídos.
**Revisável quando**: Necessidade de SSG puro ou grande equipe familiarizada com React.

## 2026-03-26: CSS e design system

**Decisão**: `TailwindCSS 4.x` sem `tailwind.config.js` + cores em OKLCH
**Motivo**: Tailwind v4 usa configuração via `@theme` no CSS — mais simples, sem arquivo extra. OKLCH garante consistência perceptual das cores (mesmo L = mesmo peso visual).
**Alternativas descartadas**:
- Tailwind v3 com config.js — versão anterior, sem os benefícios do v4
- CSS Modules — mais verboso, sem utilities prontas
- Styled Components — incompatível com filosofia do projeto
**Trade-off aceito**: Menos tutoriais disponíveis para Tailwind v4; docs oficiais são suficientes.
**Revisável quando**: Mudança de framework principal.

## 2026-03-26: Banco de dados e ORM

**Decisão**: `Neon PostgreSQL 17` + `Drizzle ORM`
**Motivo**: Neon oferece serverless com branching para dev/staging sem custo adicional. Drizzle é type-safe, próximo ao SQL e sem "magia" de abstração — mais fácil de depurar.
**Alternativas descartadas**:
- Supabase — mais opinionado, auth acoplado dificulta customização
- PlanetScale — MySQL, não PostgreSQL
- Prisma — mais abstrato, geração de tipos mais lenta, menos controle sobre queries
**Trade-off aceito**: Drizzle tem ecossistema menor que Prisma; documentação ainda crescendo.
**Revisável quando**: Volume de dados exigir PostgreSQL dedicado (DigitalOcean/RDS).

## 2026-03-26: Chave primária padrão

**Decisão**: `id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY`
**Motivo**: Padrão SQL moderno, sem dependência de sequências externas, compatível com Neon/PostgreSQL 17, mais explícito que SERIAL.
**Alternativas descartadas**:
- `SERIAL` — syntax sugar deprecated implicitamente; menos explícito
- `UUID` — overhead de índice maior, sem benefício real em projetos municipais de volume moderado
**Trade-off aceito**: IDs sequenciais são previsíveis (aceitável em sistemas internos).
**Revisável quando**: Necessidade de IDs distribuídos ou merge entre múltiplos bancos.

## 2026-03-26: Autenticação

**Decisão**: `Neon Auth (Better Auth)` como provider primário, `Google OAuth fechado` como fallback, intercambiáveis via `AUTH_PROVIDER` no `.env`
**Motivo**: Zero reescrita ao trocar de provider. Controle total de quem acessa — ninguém entra sem estar pré-cadastrado, independente do provider.
**Alternativas descartadas**:
- Auth.js (NextAuth) — acoplado ao ecossistema Next.js
- Lucia — descontinuado
- JWT próprio — manutenção e segurança são responsabilidade do projeto
**Trade-off aceito**: Duas implementações para manter; compensado pela flexibilidade operacional.
**Revisável quando**: Um provider atender 100% dos casos sem necessidade de fallback.

## 2026-03-26: Deploy

**Decisão**: `Vercel`
**Motivo**: Zero configuração para SvelteKit, integração nativa com Neon, preview deployments automáticos por branch, free tier suficiente para sistemas municipais internos.
**Alternativas descartadas**:
- DigitalOcean App Platform — mais configuração, sem preview deployments gratuitos
- Cloudflare Pages — edge runtime tem limitações com Drizzle/Neon em alguns cenários
- VPS próprio — manutenção de infra foge do escopo do projeto
**Trade-off aceito**: Vendor lock-in na Vercel; mitigado pelo adapter SvelteKit (troca é possível).
**Revisável quando**: Requisito de dados em território nacional (LGPD) ou custo justificar migração.

## 2026-03-26: Gráficos

**Decisão**: `Apache eCharts` como biblioteca obrigatória de gráficos
**Motivo**: Maior variedade de tipos de gráfico, melhor suporte a dados complexos (séries temporais, pirâmides etárias, mapas de calor), API declarativa consistente.
**Alternativas descartadas**:
- Chart.js — menos tipos, API mais imperativa
- Recharts — acoplado ao React
- D3.js — baixo nível, curva de aprendizado alta sem ganho proporcional
**Trade-off aceito**: Bundle maior que Chart.js; irrelevante para sistemas internos.
**Revisável quando**: Tecnicamente impossível implementar um gráfico específico com eCharts.

## 2026-03-26: Conventional Commits

**Decisão**: Conventional Commits sem ferramentas de enforcement (apenas convenção)
**Motivo**: Histórico legível sem overhead de configuração. Projetos solo/pequenos não precisam de husky/commitlint para se beneficiar da convenção.
**Alternativas descartadas**:
- Commitlint + Husky — overhead desnecessário para o tamanho atual dos projetos
- Mensagens livres — histórico ilegível após poucos meses
**Trade-off aceito**: Sem enforcement automático — depende de disciplina.
**Revisável quando**: Time crescer e inconsistências virarem problema real.

## 2026-03-28: Modelagem do banco de dados — schema protese

**Decisão**: 15 tabelas no schema `protese`; referência a `shared.estabelecimentos` sem alterar o schema compartilhado.
**Motivo**: O schema `shared` é consumido por múltiplos projetos (PSE, CER etc.). Adicionar colunas específicas do fluxo de próteses contaminaria o schema compartilhado com regras de negócio de um só sistema.
**Alternativas descartadas**:
- Adicionar `responsible_unit_id` em `shared.estabelecimentos` — acoplamento indevido de regra de negócio específica no schema compartilhado
- Criar `protese.health_units` com FK para `shared.estabelecimentos` — duplicação desnecessária; os IDs de `shared.estabelecimentos` já são estáveis
**Trade-off aceito**: Queries de visibilidade por unidade exigem JOIN com `protese.unit_responsibilities`; compensa pela separação limpa de responsabilidades.
**Revisável quando**: O conceito de unidade responsável for adotado por outros projetos no workspace — aí faz sentido migrar para `shared`.

## 2026-03-28: Observações como tabela de histórico (referral_notes)

**Decisão**: `protese.referral_notes` — tabela append-only, uma linha por observação
**Motivo**: Dentistas, atendentes e coordenadores inserem observações em momentos diferentes do processo. Um campo `text` único apagaria o histórico e tornaria impossível saber quem escreveu o quê — crítico para auditoria de ouvidoria.
**Alternativas descartadas**:
- Campo `notes text` em `referrals` — sem rastreabilidade de autor e timestamp por entrada
**Trade-off aceito**: Interface precisa renderizar uma lista de notas em vez de um campo único; ligeiramente mais complexo no frontend.
**Revisável quando**: Nunca — rastreabilidade em saúde pública é requisito não-funcional.

## 2026-03-28: Custódia da prótese em colunas de appointments

**Decisão**: Campos `prosthesis_ready_at/by` e `prosthesis_received_at/by` diretamente em `appointments`
**Motivo**: O fluxo de custódia tem exatamente dois atores e dois momentos por consulta. Uma tabela separada (`prosthesis_custody`) adicionaria um JOIN em todas as queries de agenda sem ganho expressivo.
**Alternativas descartadas**:
- Tabela `protese.prosthesis_custody` — JOIN extra em toda query de agenda; não justificado pela complexidade
**Trade-off aceito**: Colunas ficam NULL para a 1ª consulta (escaneamento); aceitável dado o ganho de simplicidade.
**Revisável quando**: O fluxo de custódia ganhar etapas adicionais (ex: controle de remessa, nota de devolução).

## 2026-03-28: Schema modular (schema/ com 6 arquivos)

**Decisão**: `src/lib/server/db/schema/` com 6 arquivos por domínio + `index.ts` + `relations.ts`
**Motivo**: 15 tabelas em um único `schema.ts` ultrapassa 500 linhas — difícil de navegar e de revisar em PR. Separação por domínio (users, referrals, appointments, quality, config, shared) reflete o fluxo do Discovery.
**Alternativas descartadas**:
- `schema.ts` único — convenção do template CLAUDE.md, mas inadequado para este volume
**Trade-off aceito**: Mais arquivos para abrir; mitigado pelo `index.ts` como ponto único de importação.
**Revisável quando**: Nunca há motivo para regredir para arquivo único neste tamanho.

## 2026-04-02: consecutive_misses — cálculo em runtime

**Decisão**: Calcular faltas consecutivas em runtime via query de outcomes de `appointments`, ordenados por `scheduled_date DESC`, com streak calculado em JS no `load` da fila.
**Motivo**: Volume baixo (sistema municipal); campo desnormalizado exigiria atualização a cada registro de outcome e criaria risco de inconsistência — complexidade sem ganho real.
**Alternativas descartadas**:
- Campo `consecutive_misses` em `appointments` ou `referrals` — desnormalização desnecessária; sync manual propenso a bugs
**Trade-off aceito**: Query adicional por carregamento da fila; irrelevante para o volume esperado.
**Revisável quando**: Volume crescer a ponto de a query impactar latência perceptível (improvável neste domínio).

## 2026-04-01: Estratégia de migração de schema — ALTER TABLE direto via MCP

**Decisão**: Mudanças de schema são aplicadas com `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` via MCP Neon, nunca via `pnpm db:push`.
**Motivo**: O banco `saude_workspace` contém tabelas legadas em português (`pacientes`, `consultas`, `semanas`, `finalizados`, etc.) de uma versão anterior do sistema que foi reescrita. O Drizzle desconhece essas tabelas — ao rodar `db:push`, compara o schema TypeScript com o banco real e propõe `DROP SCHEMA protese` (26 tabelas) para recriar tudo do zero. Isso causaria perda total de dados.
**Alternativas descartadas**:
- `pnpm db:push` — propõe DROP destrutivo; perigoso enquanto houver tabelas legadas no schema
- `pnpm db:migrate` com o `0000_` gerado — tentaria `CREATE TABLE` em tabelas que já existem; quebraria com erro "relation already exists"
- Limpar tabelas legadas primeiro — possível no futuro, mas requer confirmação explícita de que não há dados relevantes nelas
**Trade-off aceito**: Drizzle Kit perde o papel de gerenciador de migrations neste projeto por enquanto; compensado pela segurança total dos dados e pelo acesso direto via MCP.
**Revisável quando**: As tabelas legadas forem removidas do banco e o `0000_` for marcado como aplicado na tabela `__drizzle_migrations`, permitindo retomar o fluxo normal de `db:generate` + `db:migrate`.
